Function Chaining - The Assembly Line Pattern

Concept: Functions work together like stations in an assembly line

Input Data → Function 1 → Function 2 → Function 3 → Final Result
Function Chaining Example: # Each function does ONE job and passes result to next domain = get_user_input() # Step 1: Get data risk_level = assess_risk(domain) # Step 2: Process data action = decide_action(risk_level) # Step 3: Make decision display_result(domain, action) # Step 4: Show result Key Rules: • Return value from Function 1 → Parameter for Function 2 • Each function has ONE clear responsibility • Order matters - call functions in correct sequence • Store return values in variables for next function

Sets - Collections of Unique Items

Creating Sets: # Method 1: Curly braces threats = {"spam.com", "malware.net", "phishing.org"} # Method 2: set() function safe_domains = set(["google.com", "etsu.edu", "github.com"]) # Method 3: From list (removes duplicates) mixed = ["apple.com", "google.com", "apple.com"] unique = set(mixed) # {'apple.com', 'google.com'} Basic Set Operations: threats.add("new-threat.com") # Add item threats.remove("spam.com") # Remove (error if not found) threats.discard("maybe-missing.com") # Safe remove "malware.net" in threats # Check membership → True/False len(threats) # Count items

Set Operations Examples

Email Security Scenario: fbi_threats = {"threat1.com", "spam.net", "malware.org"} vendor_list = {"spam.net", "adware.biz", "malware.org"} our_blocks = {"spam.net", "old-threat.com"} # Union: All possible threats to consider all_threats = fbi_threats | vendor_list # Result: {"threat1.com", "spam.net", "malware.org", "adware.biz"} # Intersection: High-confidence threats (multiple sources agree) confirmed = fbi_threats & vendor_list # Result: {"spam.net", "malware.org"} # Difference: What we're missing from FBI recommendations gaps = fbi_threats - our_blocks # Result: {"threat1.com", "malware.org"} # Subset: Are we compliant with FBI requirements? fbi_compliant = fbi_threats <= our_blocks # Result: False (we're missing some FBI threats)

Common Patterns and Best Practices

✅ Good Function Design

def assess_domain_risk(domain): """Single clear purpose""" if "malware" in domain: return "HIGH" return "LOW" def make_decision(risk_level): """One job: decide action""" if risk_level == "HIGH": return "BLOCK" return "ALLOW" # Chain them together risk = assess_domain_risk("bad.com") action = make_decision(risk)

❌ Avoid These Patterns

def do_everything(domain): """TOO MANY responsibilities""" risk = assess_risk(domain) action = make_decision(risk) print(f"Blocking {domain}") send_alert_email() update_database() # Don't do this! # Using return value before it exists action = make_decision(risk) # Error! risk = assess_domain_risk("test.com")

Quick Reference - Set vs List vs Dictionary

Data Type Duplicates? Ordered? Best For Example
Set No No Unique collections, membership testing {"spam.com", "malware.net"}
List Yes Yes Sequences, ordered data ["first", "second", "first"]
Dictionary No (keys) Yes (3.7+) Key-value pairs, lookups {"domain": "risk_level"}

Debugging Tips

Function Chain Debugging: # Print intermediate results to see data flow domain = get_domain() print(f"DEBUG: Got domain: {domain}") risk = assess_risk(domain) print(f"DEBUG: Risk level: {risk}") action = decide_action(risk) print(f"DEBUG: Final action: {action}") Set Operation Debugging: # Check set contents and sizes print(f"Set A has {len(set_a)} items: {set_a}") print(f"Set B has {len(set_b)} items: {set_b}") print(f"Intersection: {set_a & set_b}") print(f"Union size: {len(set_a | set_b)}")

Remember: Functions are like assembly line stations - each does one job well. Sets are like bags of unique items - perfect for "who's in the group" questions!
